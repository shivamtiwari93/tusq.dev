## PM Challenges to the Intent

### Challenge 1: "Migrate docs" — but there are no user-facing docs yet

The intent says "migrate the current website/docs/blog surface." In reality:
- **Website:** A single static HTML landing page (`websites/index.html`) — this migrates directly.
- **Docs:** No user-facing documentation exists. The `.planning/` directory contains internal planning artifacts (SYSTEM_SPEC.md, command-surface.md, etc.) which are not user-facing docs. We need to **create** docs from these sources, not migrate them.
- **Blog:** No blog exists. We need to **create** a blog and adapt the ANNOUNCEMENT.md and RELEASE_NOTES.md into blog posts.

**Decision:** The scope is correctly framed as "create a docs platform" with content derived from accepted artifacts, not a mechanical migration.

### Challenge 2: Content must not drift from accepted product truth

The prior run produced carefully scoped MESSAGING.md with explicit "Claims We Can Defend" and "Claims We Must Not Make" sections. The Docusaurus content must stay within these boundaries. The SYSTEM_SPEC includes a dedicated M7 product truth audit milestone to enforce this.

**Decision:** Product truth alignment is a first-class acceptance criterion, not a nice-to-have.

### Challenge 3: Directory naming — `website/` vs `websites/`

The existing static site lives in `websites/`. Docusaurus convention is `website/` (singular). We should use `website/` for the Docusaurus project and leave `websites/` in place until the migration is confirmed working, then remove it.

**Decision:** New Docusaurus project goes in `website/`. Old `websites/` is not deleted during this initiative — that's a cleanup task for after verification.

### Challenge 4: Scope creep risk — fancy features

Docusaurus supports search, i18n, versioned docs, custom plugins, and many other features. None of these are needed for the initial platform. The spec explicitly excludes them.

**Decision:** Vanilla Docusaurus 3.x with default plugins only. Homepage swizzling is the only customization.

### Challenge 5: Sensitivity class — ship the field or defer entirely?

The vision (VISION.md line 57) lists "side effects and sensitivity class" as part of the canonical artifact. Side effects are implemented in V1 with `classifySideEffect()`. Sensitivity is not implemented at all — no code, no field, no inference.

**Options considered:**
- **A) Defer entirely to V2.** No field in V1 manifest. Simpler, but changes the artifact shape later, breaking consumers.
- **B) Add field with `"unknown"` default.** Field exists, no inference. Mirrors the `input_schema`/`output_schema` pattern. Consumers can rely on the field being present. Humans can manually edit it during manifest review.
- **C) Add field with basic inference from auth_hints.** More useful but adds implementation complexity and risk of false classification.

**Decision:** Option B. The `sensitivity_class` field ships in V1 as `"unknown"` for all capabilities. This establishes the contract without overclaiming inference accuracy. It follows the exact same pattern as V1 schemas: structurally present, intentionally conservative, manually editable, with real inference deferred to V2.

### Challenge 6: Side effect classification — should `financial` be a separate class?

The vision (VISION.md line 160) mentions "financial" as a separate category: "read, write, destructive, financial, or security-sensitive." The current V1 implementation uses only three values: `read`, `write`, `destructive`.

**Decision:** Keep three values for `side_effect_class`. Financial and security-sensitive are **sensitivity concerns**, not side-effect concerns. A `POST /payments` is a `write` (side effect) that is `confidential` or `restricted` (sensitivity). Conflating mutation type with data sensitivity in one field would create ambiguity. The two-field model (`side_effect_class` + `sensitivity_class`) is cleaner and matches how agent runtimes actually make decisions: "can I call this without confirmation?" (side effect) vs. "what audit/redaction rules apply?" (sensitivity).

### Challenge 10: Redaction vs sensitivity — redundant or complementary?

With `sensitivity_class` already classifying data types (public/internal/confidential/restricted) and `redaction` specifying masking rules, there is a question of whether these overlap.

**Decision:** They are complementary, not redundant. Sensitivity is the *classification* (what kind of data); redaction is the *operational response* (what to do about it). A capability can be `confidential` (touches PII) but the team may choose different redaction policies: one team masks emails but logs everything else, another team goes fully silent. Sensitivity drives access decisions; redaction drives logging/audit decisions. Keeping them separate preserves policy flexibility.

### Challenge 13: Capability digest — what fields to include?

### Challenge 16: Manifests are not usable on first pass for real codebases

VISION.md line 72: "produce manifests that are usable on first pass for real codebases, not toy examples." Line 73: "treat manual manifest authoring as a failure of the engine, not a feature." Prior turns (DEC-143 through DEC-223) systematically addressed all seven governance metadata dimensions. But none addressed whether the core extraction output — the fields an LLM actually reads to understand a tool — is usable without manual enrichment.

**Current state of a first-pass manifest for a real codebase:**

| Field | What it produces | Usable? |
|-------|-----------------|---------|
| `name` | `get_api_api_v1_users_id` | No — domain `api` is wrong, path slug is noisy |
| `description` | "GET /api/v1/users/:id capability in api domain" | No — tells an LLM nothing about what the tool does |
| `input_schema` | `{ type: "object", additionalProperties: true }` | No — LLM has no idea what parameters to send, even though `:id` is in the path |
| `domain` | `api` (from `/api/v1/users/:id`) | No — should be `users` |
| `confidence` | 0.86 (named handler + auth hints) | Misleading — suggests route is well-understood when schema is empty |

**The four deficiencies are fixable in V1** because they use only information the scanner already has:

1. **Path parameter extraction.** The route path `/users/:id` already contains the parameter name. Regex extraction of `:param` and `{param}` tokens and injecting them as `input_schema.properties` with `required: true` is trivial. This makes the difference between an LLM that knows it needs to provide `id` and one that guesses.

2. **Smart domain inference.** The path segments are already split. Skipping well-known API prefixes (`api`, `v1`, `v2`, `v3`, `rest`) before taking the first meaningful segment turns `/api/v1/users/:id` → domain `users`. Zero new parsing required.

3. **Rich capability descriptions.** The handler name, auth hints, side effect class, and method/path are all already computed before `describeCapability()` runs. Using them produces "Retrieve a specific user by ID — read-only, requires authentication (handler: getUser)" instead of "GET /api/v1/users/:id capability in api domain." The LLM gets semantic context.

4. **Honest confidence penalty for schema-less routes.** A route scoring 0.86 with `additionalProperties: true` and no real schema is misleading. Adding a penalty of -0.10 when no schema hint is detected (bringing such routes to 0.76, below the `review_needed` threshold of 0.8) ensures the manifest honestly flags routes where the engine couldn't extract structure.

**Options considered:**

- **A) Defer all four to V2.** Leaves manifests unusable on first pass for an indefinite period. Violates VISION line 72 and 73.
- **B) Implement path params only.** Lowest effort but leaves descriptions useless and domain wrong.
- **C) Implement all four.** Each is a small, contained change in `src/cli.js`. Combined, they transform a manifest from "requires full manual rewrite" to "requires targeted review and enrichment."

**Decision:** Option C. All four improvements use only data the scanner already computes. They transform first-pass manifest quality from "structurally present but functionally empty" to "actionable with targeted review." This is the minimum bar for VISION line 72.

### Challenge 18: M20 — Opt-in local execution policy scaffold, not an execution engine

On 2026-04-22 the approved intake intent `intent_1776835221317_7398` requested "an opt-in local execution-policy scaffold for the MCP serve path. The goal is to move beyond describe-only toward safe execution wrappers without live product side effects." The prior commit 71c4abc recovered intake state for M20 but did **not** extend the four required planning artifacts — ROADMAP.md still stopped at M19, SYSTEM_SPEC.md had no execution-policy section, command-surface.md had no `tusq serve --policy` surface, and this PM_SIGNOFF had no M20 challenge. The `planning_signoff` gate cannot pass while the planning artifacts describe only M1–M19.

**The real risk of this increment is not scope ambiguity — it is scope drift into live execution.** VISION.md lines 177–181 ladder dry-run → confirm → approve → execute. It is tempting to collapse dry-run and confirm into a single "try it and see" CLI. That would violate VISION.md line 180 ("make risky operations explicit and governable") and the M20 intent ("Keep all work repo-local/offline and governed").

**Decision:** M20 ships exactly one rung of the ladder: **dry-run argument validation plus auditable plan emission**. Concretely:

1. **Opt-in by a file, not by a flag default.** A new `.tusq/execution-policy.json` file is the operator-authored artifact. Without it, `tusq serve` is byte-for-byte identical to V1 describe-only. The policy file is a governance artifact, reviewable the same way a manifest is reviewable.
2. **`tusq serve --policy <path>` is the only new CLI surface.** No `tusq execute`, no `tusq confirm`, no `--live`. Adding those commands would imply V1.1 can execute, which it cannot.
3. **`tools/call` under `mode: "dry-run"` validates arguments against the approved compiled tool's `parameters` schema and returns a structured `dry_run_plan` with an explicit `executed: false` marker.** Every response carries `executed: false`. Downstream consumers cannot mistake a plan for an execution.
4. **Approval gate invariants are preserved, not relaxed.** Only capabilities with `approved: true` appear in `tools/list`. `allowed_capabilities` is a strict subset filter on top of the approval gate, never a replacement. Dry-run mode never bypasses approval.
5. **No outbound I/O of any kind to the target product.** Not even a HEAD probe, not even a DNS lookup. The scaffold runs entirely in-process and emits a plan, not a request.
6. **Validation depth is intentionally narrow for V1.1.** Required fields, primitive type checks, and `additionalProperties: false` rejection. No JSON Schema format/enum/oneOf/regex validation. This is enough to catch the arguments-an-LLM-guesses bug class without committing to a schema validator dependency.

This decision consciously rejects three tempting alternatives:

- **A) A hosted sandbox that forwards requests to a test environment.** Violates the local/offline contract and multiplies the attack surface. Rejected.
- **B) A `tusq execute --confirm` CLI that prompts before live execution.** Crosses the describe-only boundary. Rejected for V1.1; reserved for a later increment once V1.1 dry-run has soaked.
- **C) Full JSON Schema draft-07 validation with format/enum/oneOf support.** Pulls in an external validator (ajv) or reimplements one. Rejected as scope creep; the narrow four-rule validator covers the argument failure modes LLMs actually produce.

### Challenge 19: `plan_hash` is the first replay primitive — spec it now or regret it later

The dry-run plan is the first artifact on the serve path that is deterministic across invocations. Every future increment on the VISION ladder (confirm, approve, execute, replay, eval) will want to reference a plan by identity. If we ship dry-run without a stable identifier, V2 will need a breaking shape change to add one.

**Decision:** Ship `plan_hash` in the first dry-run response. The hash is a SHA-256 over a canonical JSON serialization of `{method, path, path_params, query, body, headers}` — the *request content*, not the governance echo fields. Two calls with identical validated arguments MUST produce the same `plan_hash`. This is a one-line guarantee today and the foundation for:

- **V2 confirmation.** Operator approves a `plan_hash`; execution engine refuses to execute if the plan regenerates to a different hash.
- **V2 replay.** Eval harness asserts the same arguments still produce the same plan after a manifest regeneration.
- **V2 diff.** `tusq plan-diff old.json new.json` across plan files is trivial when every plan has an identity.

The hash excludes `evaluated_at`, `policy.reviewer`, `policy.approved_at`, and `executed`. Including them would make every invocation unique and defeat the purpose.

### Challenge 21: Scaffold generator vs interactive wizard — pick the one that never blocks CI

The charter says "reduce manual setup for the opt-in local execution-policy scaffold." Three alternatives were considered:

- **A) Interactive wizard.** A `tusq policy init` that prompts the operator for mode, reviewer, and allowed capabilities. Best UX for a first-time operator at a terminal. **Rejected for V1.2** — it couples the generator to TTY state, it is hostile to CI use (where the natural next place to scaffold a policy is in a bootstrap script), and it forces us to build an interactive-input abstraction the rest of the CLI does not have.
- **B) A `--init` flag on `tusq serve`.** Cheapest to implement. **Rejected** — conflates starting a server with generating a governance artifact. Operators who run `tusq serve --init --policy new.json` would reasonably expect the server to start after generation, and smuggling two unrelated behaviors into one command violates the one-command-one-responsibility shape the rest of the CLI surface follows.
- **C) A dedicated `tusq policy init` command group (`policy` as a noun with `init` as the first verb).** Matches the naming pattern of the existing `tusq approve`, `tusq diff`, `tusq docs`, and `tusq compile` commands. **Decision:** Ship option C. The V1.2 surface is exactly one new subcommand (`tusq policy init`) under a new `policy` top-level verb. Flags are all non-interactive; CI bootstrap is a one-liner. The interactive wizard remains open as a V2 option gated on operator telemetry.

### Challenge 22: Default mode — describe-only is the only safe default

If `tusq policy init` defaults to `mode: "dry-run"`, then an operator who runs the command and then runs `tusq serve --policy .tusq/execution-policy.json` silently flips their MCP server from describe-only to dry-run plan emission. That is a quiet governance change that nobody explicitly asked for. It is the exact mistake the M20 launch artifacts (see DEC-487, DEC-488) already had to correct in the messaging layer; we should not reintroduce it in the tooling layer.

**Decision:** The default `mode` emitted by `tusq policy init` is `"describe-only"`. A dry-run policy requires the explicit `--mode dry-run` flag. The generated file then explicitly records `mode: "describe-only"` as a field, so a reviewer reading `.tusq/execution-policy.json` in a PR cannot miss that the policy is describe-only-with-extra-audit-fields rather than dry-run. This default is codified in SYSTEM_SPEC.md Constraint 8 as the "M21 safe-default invariant" so dev cannot flip it without explicit PM re-approval.

This decision consciously rejects two tempting alternatives:

- **A) Default to dry-run because "that's the interesting case."** Rejected — the "interesting case" for a first-time operator is "nothing about my serve path changed by accident." Describe-only preserves that.
- **B) Default to the mode most recently used in the repo (infer from history).** Rejected — introduces filesystem state to a command that is otherwise stateless, and invites the same drift M21 is trying to eliminate.

### Challenge 23: `tusq policy verify` vs a `--check` flag on `tusq serve` — pick the one CI can safely run

The charter says "tighten governance around execution-policy artifacts." There are three plausible ways to let CI validate a policy file without running a live MCP server:

- **A) `tusq serve --check --policy <path>` that runs validation and exits.** Cheapest to implement. **Rejected** — it conflates "start a server" with "check a file," and some CI runners would hit the TCP port-bind code path even with an early-exit flag. Smuggling two behaviors into `tusq serve` violates the one-command-one-responsibility shape the rest of the CLI follows, and it makes the gate easy to misuse (forget the `--check` flag and you accidentally start a server in CI).
- **B) Import `loadAndValidatePolicy()` in a CI script directly.** Technically possible today because the function is exported. **Rejected as a governance artifact** — it forces every team to write their own Node entry point, it exposes an internal API as a product contract, and it makes the validator message strings part of the public surface without a CLI to channel that contract through. Operators who do not write Node tooling cannot gate their pipeline at all.
- **C) A dedicated `tusq policy verify` subcommand under the `policy` noun.** Matches the naming pattern already established by `tusq policy init` in M21 and leaves room for V2 siblings (`tusq policy verify --strict`, `tusq policy diff`, `tusq policy migrate`) under the same noun without future renaming. **Selected.**

**Decision:** Ship option C. The V1.3 surface is exactly one new subcommand (`tusq policy verify`) under the existing `policy` top-level verb. The command reuses `loadAndValidatePolicy()` by function call — no re-implementation, no duplicated error strings — and the shared validator is what makes the contract enforceable.

### Challenge 24: Validator parity is the contract, not wording consistency

It is tempting to let `tusq policy verify` produce its own, slightly friendlier error messages ("Policy is invalid — the `mode` field must be one of…"). That is a wording question, and it is the wrong frame.

The real governance contract M22 ships is this: **a policy that passes `verify` must start `serve`; a policy that fails `verify` must fail `serve` startup with the same message.** If the two entry points drift in either direction, the verifier is a false oracle — a CI gate that greenlights files the server will reject, or that rejects files the server would accept. Either outcome is worse than no verifier at all, because it costs the operator trust.

**Decision:** The M22 contract is that `tusq policy verify` and `tusq serve --policy` share `loadAndValidatePolicy()` byte-for-byte, and every error message is identical across the two entry points. This is codified in SYSTEM_SPEC.md as Constraint 9 ("M22 validator-parity invariant"). A dedicated smoke parity fixture (see REQ-070+ in the docs/tests section) asserts, for every failure case, that both entry points exit 1 and print the same message. A friendlier wording layer, if ever needed, lands in the shared validator so both paths pick it up simultaneously — never in one entry point alone. This decision consciously rejects the wording-divergence path even though the friendlier messages are tempting.

### Challenge 25: V1.3 stays policy-file-only; manifest-aware `--strict` is explicitly V2

The most obvious "good idea" to add to `tusq policy verify` is a `--strict` flag that reads `tusq.manifest.json` and flags any name in `allowed_capabilities` that does not exist as an approved capability. That would catch the category of error where an operator generates a policy with `--allowed-capabilities refund,cancel` but those capabilities are not actually in the manifest, so `tusq serve --policy` silently serves zero tools.

Adding it now is the wrong move for three reasons:

1. **It breaks the local-only invariant as specified.** M22 is a *policy-file* validator. Once the verifier reads the manifest, it inherits the manifest's I/O assumptions, its version/hash semantics, and its freshness concerns. That is a different governance surface — and the right abstraction for it is its own subcommand (`tusq policy verify --strict` or `tusq policy audit`), not a quiet flag on the entry-point verifier.
2. **It couples two governance boundaries that should stay decoupled.** A policy file can be valid even when the manifest is stale, missing, or not yet generated — for example, in a repo that ships `.tusq/execution-policy.json` as a pre-seeded governance artifact before the first scan. Making `policy verify` require a manifest would break that workflow for no operator benefit in V1.3.
3. **It invites scope creep.** Once `policy verify` reads the manifest, the next obvious request is "also validate that the approved capabilities in the manifest are still syntactically valid," and the one after that is "also run the eval harness," and so on. That path ends at a `tusq ship-check` command, not a policy verifier.

**Decision:** V1.3 `tusq policy verify` does not read the manifest. Manifest-aware cross-referencing is reserved for a dedicated V2 increment (provisionally `tusq policy verify --strict`, ledgered here so the future design does not need to rediscover it). This decision is codified as SYSTEM_SPEC.md Constraint 10 ("M22 local-only invariant") so dev cannot add manifest I/O without explicit PM re-approval.

### Challenge 26: `--strict` vs a new `tusq policy audit` subcommand — pick the one that keeps the governance surface small

The M22 `Challenge 25` decision already promised `tusq policy verify --strict` as the future manifest-aware surface; M23 is cashing that ledgered promise. Even so, there were two plausible ways to ship the strict check:

- **A) New subcommand `tusq policy audit` that always reads the manifest.** Cleaner separation between "pure policy-file validator" and "policy-against-manifest auditor." **Rejected** — two verbs under `policy` that both take a policy path and both exit 0/1 would force operators to learn a second command shape for essentially the same governance question ("is this policy file safe to use with my current manifest?"). It also fragments the parity contract: a reviewer now has to know that `serve --policy` pairs with `verify`, but *not* with `audit`, and that CI should run `audit` (not `verify`) when they want manifest cross-reference. That is more governance surface than the problem justifies.
- **B) A `--strict` flag on the existing `tusq policy verify`.** The strict checks are additive — everything `verify` already rejects, `verify --strict` also rejects — and they operate on the same policy file the default `verify` reads. **Selected.**

**Decision:** Ship option B. M23 is exactly one new flag, `--strict`, on the existing `tusq policy verify` subcommand, plus one supporting flag (`--manifest <path>`) that is only consulted when `--strict` is set. No new subcommand, no new top-level verb. The V2 slot `tusq policy audit` remains open for a future multi-source governance report (policy + manifest + scan + evals), but M23 does not consume that slot.

### Challenge 27: Opt-in, not default — the verifier must not silently start reading the manifest

Making `--strict` the default (or making the default behavior of `tusq policy verify` silently escalate to manifest-aware checks whenever `tusq.manifest.json` happens to exist) would break the M22 local-only invariant in a way that is invisible to the operator. A CI job that previously passed `tusq policy verify` on a pre-manifest scaffolded policy would suddenly start failing after a `tusq manifest` run introduced the manifest file. A pre-commit hook that validated a seed policy before the first scan would suddenly require a generated manifest. Neither of those regressions has a user benefit; both cost operator trust.

**Decision:** `--strict` is strictly opt-in. Default `tusq policy verify` (no flags) MUST be byte-for-byte identical to the M22 behavior: no manifest I/O, no new failure modes, no inspection of the manifest file even if it happens to exist at the default path. ### Challenge 28: Strict PASS is a policy/manifest alignment statement — it is not a runtime safety claim

The largest foot-gun in strict-mode verification is narrative: a `tusq policy verify --strict` PASS can easily be misread as "it is now safe to execute any capability in `allowed_capabilities`." **Decision:** Codify the strict-PASS framing as SYSTEM_SPEC.md Constraint 12 ("M23 least-privilege-validation invariant") and reserve specific forbidden framings in the launch-artifact layer: strict mode must not be described as "runtime check," "execution-safety gate," "pre-flight for serve --policy," "manifest freshness validator," or any phrase that implies a PASS makes execution safe. This decision consciously rejects two tempting alternatives:

- **A) Add a freshness check (`manifest_version`, `previous_manifest_hash`).** Introduces manifest-lineage semantics that the verifier is intentionally avoiding. Reserved for a future governance ladder rung (possibly `tusq policy audit`). Rejected for V1.4.

### Challenge 31: Pick the most deterministic slice of "validator-aware input-schema inference" — Fastify `schema.body` only

The charter explicitly mentions "validator-aware input-schema inference for existing Node.js route discovery if feasible." The V1 frameworks shipped under M14 are Express, Fastify, and NestJS. Each has a canonical validator style:

- **Express + express-validator.** Chain expressions: `body('email').isEmail().normalizeEmail()`. Requires AST parsing or sophisticated chain-expression inference to extract field names and types. Library-specific; validator chaining varies across repos.
- **Fastify route-level `schema`.** A literal JSON Schema object attached directly to the route-options object: `schema: { body: { type: 'object', properties: { email: { type: 'string' } }, required: ['email'] } }`. Literal JSON Schema by convention; Fastify's own runtime compiles it via ajv.
- **NestJS + class-validator DTOs.** Decorator-based: `@IsEmail() email: string`. Requires TypeScript type-analysis plus decorator-metadata resolution. Deeply library-specific.

Only the Fastify case is deterministic under a regex + balanced-brace extractor. Express-validator chains require either a full AST parser (ts-morph, babel) or an opinionated chain-expression grammar — both are dependency-heavy and violate the local/offline / zero-new-dependency invariants this project has enforced since M14. NestJS DTOs require a TypeScript compiler-API integration that is even heavier. Both remain valuable for V2, but neither fits a ~0.5-day bounded increment that produces real code, tests, and docs without expanding the tool's attack surface or its dependency footprint.

The Fastify case is the one where the source code literally encodes the answer the manifest needs. A Fastify route declaring `body: { properties: { email: { type: 'string' } } }` is telling the scanner, in the most unambiguous possible form, "the body of this capability has a field named `email` of type string." The V1.4 scanner discards this information — it detects the presence of the `schema:` keyword via `schema_hint` but throws away the declared shape. M24 closes exactly that gap.

**Decision:** M24 ships literal Fastify `schema.body` extraction only. Express-validator chain inference and NestJS DTO inference are explicitly V2 scope, reserved for a future `tusq scan --plugin <name>` pluggable extraction path. This decision is codified as SYSTEM_SPEC.md Constraint 13 ("M24 static-literal-extraction invariant").

This decision consciously rejects three tempting alternatives:

- **A) Ship all three validator styles in one increment.** Three distinct extraction paths, three dependency evaluations, three failure surfaces, three QA surfaces — ~0.5 day becomes ~3 days and the increment stops being reviewable. Rejected.
- **B) Ship Fastify + a thin express-validator chain extractor.** The "thin" extractor either handles 30% of express-validator patterns (creating a long tail of unsupported cases reviewers have to remember) or requires a real chain-expression parser (violates the zero-new-dependency invariant). Rejected for V1.5.
- **C) Defer the whole increment; declare V1.4 "done" and ship nothing new.** Violates the charter ("derive and execute the next VISION-aligned increment") and leaves a concrete, addressable VISION.md line-72 gap unfilled when the fix is in the deterministic zone. Rejected.

### Challenge 32: Opt-in on pattern match, not on a flag — add the inference silently when the source is unambiguous

A plausible alternative to "detect a literal Fastify schema and extract its fields" is "add a `tusq scan --extract-fastify-schemas` flag that operators opt into." That framing sounds conservative — operators control when the new inference runs. But it introduces three problems:

1. **Two shapes for the same repo.** A manifest generated with the flag and one without would drift on every regeneration. M13 `capability_digest` would flip based on flag state, not on source reality. Re-approval would depend on CLI invocation, not on capability change.
2. **Surface accretion.** Every subsequent validator-aware extractor (express-validator, NestJS DTOs, Zod) would need its own flag, and operators would have to remember which flags align with which libraries in their codebase. The V1 CLI surface is already 12 commands; adding 5+ scanner flags bloats the governance model for no operator benefit.
3. **Missing the point.** Fastify's `schema.body` is literally the source telling the scanner the answer. Treating it as opt-in inference is akin to treating path parameters as opt-in — M15 correctly made path-parameter extraction the default because the source is unambiguous. M24 applies the same principle.

The one real risk is silent manifest drift: an existing Fastify codebase with declared schemas would, on first scan post-M24, emit a different `input_schema` shape. M13 `capability_digest` will flip and re-approval will be required — that is exactly the diff-and-approve workflow M13 ships for. The drift is neither silent to a reviewer (it surfaces in `tusq diff`) nor unsafe (the approval gate holds — `tools/list` continues to filter by `approved: true`).

**Decision:** M24 extraction is active by default on pattern match. No flag. The opt-in is the source itself: a route that does not declare a literal Fastify `schema.body` block gets M15 fall-back behavior byte-for-byte. Operators who want to suppress the inference can either remove the schema block from source (which they would not, because it is authoritative) or simply never re-approve the updated capability (which correctly holds the change behind the M13 approval gate). This decision is codified as SYSTEM_SPEC.md Constraint 13; the source-literal framing is Constraint 14.

This decision consciously rejects two tempting alternatives:

- **A) Ship a `--extract-schemas` flag defaulting off.** Creates two competing manifest shapes for the same repo and invites operators to forget the flag. Rejected.
- **B) Ship the extraction behind an environment variable (`TUSQ_FASTIFY_SCHEMA_EXTRACTION=1`).** Worse than A — environment state is invisible in the manifest itself, so a reviewer cannot tell which shape was generated under which regime. Rejected.

### Challenge 33: Do NOT claim the extracted shape is "validator-backed" — it is source-literal only

The largest narrative risk M24 ships is framing. The words "validator-aware," "validator-backed," "runtime-validated," "ajv-validated" all sound like the manifest now proves something about runtime behavior. They do not. M24 reads the literal bytes of the source file and extracts the declared JSON Schema object. Whether Fastify's runtime actually enforces that schema, whether ajv is actually installed, whether the route registration is reachable — none of that is proven by the manifest. A reviewer who reads `input_schema.source: "fastify_schema_body"` and concludes "Fastify will reject any request that violates this shape" is reading more into the manifest than the scanner observed.

This is the same failure mode M23's Constraint 12 guards against (strict PASS ≠ runtime safety), applied to a different surface. The defensible framing for M24 is "the declared body schema as it appears literally in source" — one step less than runtime enforcement, one step more than heuristic.

**Decision:** Codify the source-literal framing as SYSTEM_SPEC.md Constraint 14 ("M24 source-literal-framing invariant"). Reserve specific forbidden framings in docs, README, CLI help, and launch artifacts: the extracted shape must not be described as "validator-backed," "runtime-validated," "ajv-validated," "Fastify-enforced," or any phrase that implies runtime enforcement. The only defensible framing is "the declared body schema as it appears literally in source" and, by extension, "higher-signal than a keyword hint, lower-signal than runtime enforcement." Every M24 doc/README/CLI-help/launch artifact inherits this constraint.

This decision consciously rejects two tempting alternatives:

- **A) Invoke ajv at scan time to validate that the extracted schema is itself well-formed JSON Schema.** Crosses the static-extraction boundary; introduces an ajv dependency; violates Constraint 13. Rejected.
- **B) Add a runtime probe that asks the target Fastify server whether the schema is active.** Violates the local/offline invariant. Rejected outright.

### Challenge 34: Pick the narrowest deterministic slice of "sensitive-field detection" — field-name match only, frozen canonical set

The charter explicitly asked for a "bounded static sensitive-field/redaction increment that improves `tusq.manifest.json` usefulness for real SaaS codebases while preserving local/offline deterministic behavior." VISION.md lines 167–168 name four workstreams under data-and-schema understanding: detect PII, detect payments, detect secrets, detect regulated data, and produce redaction/retention defaults. Each of those could be implemented at several depths:

- **A) Static field-name matching against a frozen canonical PII set.** Pure function over `input_schema.properties`; no new dependency; deterministic; whole-key match; overclaim risk is bounded because the only claim is "field key matches a well-known PII name."
- **B) Static name match + format/regex inference (detect emails by pattern, credit cards by Luhn, etc.) at scan time.** Requires building a validator registry; expands the claim space (now the manifest implies the field *looks like* an email, not just that its *name* matches); adds non-trivial false-positive surface; increases test scope to several hundred regex cases.
- **C) Runtime payload sampling.** Violates the local/offline invariant. Out of scope.
- **D) Full regulatory inference (GDPR/HIPAA/PCI mapping).** Requires a taxonomy, a jurisdiction model, and a policy engine. At least a multi-week increment. Out of scope for a ~0.5-day bounded increment.
- **E) ORM/DB-column-name redaction.** Requires an ORM model parser (TypeORM, Prisma, Sequelize, Mongoose) that is library-specific and AST-heavy. Deferred as V2 plugin scope.
- **F) Multi-locale name support (Spanish, German, Portuguese, Japanese, etc.).** Requires a localization dictionary (new dependency) or locale detection heuristic. Deferred as V2.

Only (A) fits all four bounding constraints (deterministic, local/offline, zero new dependency, ~0.5-day scope) and still produces visible VISION-aligned value for real Node.js SaaS codebases. Option (A) builds directly on M15 + M24: those two increments already populated `input_schema.properties` with real declared field names, so (A) is a pure function over an already-built object. The implementation is reviewable in under 50 lines of `src/cli.js`; the test surface is a fixed enumeration.

**Decision:** M25 ships whole-key case-insensitive PII field-name matching against a frozen canonical set. Options (B), (C), (D), (E), and (F) are explicitly V2 scope. This decision is codified as SYSTEM_SPEC.md Constraint 15 ("M25 static-name-matching invariant").

This decision consciously rejects four tempting alternatives:

- **A) Ship name matching + format regex inference in one increment.** Doubles the claim surface (now we're inferring semantics from values as well as names); adds a new failure mode (a field named `random_token` that doesn't match canonical names but regex-matches a UUID would produce an ambiguous signal). Rejected for V1.6.
- **B) Ship a flag-gated opt-in (`tusq manifest --detect-pii`).** Creates two manifest shapes for the same repo, invites operators to forget the flag, and inconsistent `capability_digest` computation. Same reasoning as M24 Challenge 32. Rejected.
- **C) Ship multi-locale PII names in the same increment.** Requires a localization dictionary or new dependency; expands the canonical set by an order of magnitude; increases regression surface well beyond a ~0.5-day bounded increment. Rejected for V1.6.
- **D) Ship auto-escalation of `sensitivity_class` alongside PII detection.** Sensitivity is a composite judgment (data class + regulatory scope + organizational posture); field-name match alone does not prove it. Auto-escalating would silently reclassify approved capabilities on M25 adoption. Rejected explicitly; `sensitivity_class` inference is reserved for a future dedicated increment. Codified as Constraint 16.

### Challenge 35: Whole-key match, not tail-match or substring — the spec must forbid `email_template_id` from matching `email`

The most common failure mode for naive PII-field detection is substring or tail-matching. A codebase with a field named `email_template_id` (a foreign-key reference to an email template, not PII) would be flagged as PII under any substring-match rule. Same problem with `phone_book_url` (a URL to a phonebook resource), `ssn_document_uri` (a URI pointing to a Social Security document, but not the SSN itself), or `password_policy_version` (an integer version of a policy document, not a password). Every false positive on M25 adoption becomes an M13 `capability_digest` flip and a manual re-approval — per-capability reviewer cost scales with the false-positive rate.

The conservative rule is whole-key match after normalization: `key.toLowerCase().replace(/[_-]/g, '')` must equal one of the canonical names *exactly*, not contain it. This correctly flags `user_email`, `userEmail`, `USER_EMAIL`, and `user-email` (all normalize to `useremail`) while correctly not flagging `email_template_id` (normalizes to `emailtemplateid`, not in the canonical set). Dev and QA should both be able to reason about the match rule on first read.

**Decision:** M25 matching is whole-key only. Substring and tail-matching are forbidden. Case / underscore / hyphen variants all normalize to the same form and match identically. This is codified as SYSTEM_SPEC.md Constraint 15 alongside the broader static-name-matching invariant.

This decision consciously rejects two tempting alternatives:

- **A) Tail-match (`key.endsWith("_email")` flags `user_email`, `account_email`, etc.).** Would catch marginally more true positives but also catch false positives like `template_email` (a template type) and `from_email` (a sender address field in a mail route — arguably PII, but only in specific contexts). The false-positive tax on the M13 approval flow would outweigh the true-positive gain. Rejected.
- **B) Substring-match (`key.includes("email")`).** Worst option — catches `email_template_id`, `unsubscribe_email_token_id`, `email_template_body_html`, none of which are PII. Rejected outright.

### Challenge 36: Do NOT auto-escalate `sensitivity_class` on PII match — two signals, two owners

A seemingly obvious post-M25 improvement is to auto-escalate `capability.sensitivity_class` from `"unknown"` to `"confidential"` (or `"restricted"`) whenever `redaction.pii_fields` is non-empty. The intuition is valid on its face: if the capability handles PII, its data class is at least confidential. But that reasoning conflates two independent governance signals:

- `redaction.pii_fields` is a **field-level** hint: which keys in the request body or path carry PII-named data.
- `sensitivity_class` is a **capability-level** judgment: composite of data class, regulatory scope, organizational posture, and deployment context.

A route that accepts `email` as a free-text search-contact field in a CRM is genuinely confidential-sensitive. A route that accepts `email` as the sender address in an internal template-editor might be `internal` or `unknown` depending on who can call it. Auto-escalation cannot tell these apart without a richer signal. Worse, auto-escalation would silently reclassify every previously-approved capability that happens to have a PII-named body field, flipping both `redaction.pii_fields` and `sensitivity_class` in the same M25 upgrade — dragging two governance dimensions across the re-approval threshold when only one is actually supported by the extracted evidence.

**Decision:** M25 does NOT auto-escalate `capability.sensitivity_class`. It remains `"unknown"` until an explicit V2 sensitivity-inference increment ships with its own constraint entry. M25 populates `redaction.pii_fields` only. This is codified as SYSTEM_SPEC.md Constraint 16 ("M25 redaction-framing invariant") alongside the broader framing guardrails.

This decision consciously rejects three tempting alternatives:

- **A) Auto-escalate to `"confidential"` when `pii_fields` is non-empty.** Conflates field-level and capability-level signals; silently reclassifies approved capabilities; over-claims on what name matching proves. Rejected.
- **B) Auto-escalate only for the canonical "Secrets" and "Payment" categories (passwords, API keys, credit cards).** Still conflates field and capability signals; the `password` field on a 2FA toggle route is arguably not confidential in the same sense as `password` on a login route. Rejected for V1.6.
- **C) Emit a soft `sensitivity_hint: "pii_detected"` field alongside `sensitivity_class`.** Introduces a new field shape, forks the governance vocabulary, and adds a migration surface for consumers of the manifest. Sensitivity inference deserves its own increment, not a backdoor under M25. Rejected.

### Challenge 37: Move toward retention defaults without pretending a field-name category is an org policy

M26 exists because M25 made redaction hints useful but flat. A reviewer can now see `redaction.pii_fields: ["email", "password"]`, but the manifest does not distinguish contact PII from secrets. VISION.md asks for redaction and retention defaults, and the obvious temptation is to jump directly from field names to `retention_days`, `log_level`, and `mask_in_traces` defaults. That would be premature: retention and logging posture are organization-specific policy choices, not facts the scanner can infer from a source-declared field key.

The bounded step is to add category labels only: `redaction.pii_categories` mirrors `redaction.pii_fields` one-for-one and labels each matched field with a frozen category (`email`, `phone`, `government_id`, `name`, `address`, `date_of_birth`, `payment`, `secrets`, or `network`). This gives reviewers retention-relevant evidence without silently setting retention policy. The extractor remains pure, local, deterministic, and zero-dependency; it operates on M25's own output rather than rescanning source or inspecting values.

**Decision:** M26 ships static PII category labels, not retention-policy enforcement. M26 does not derive or mutate `redaction.retention_days`, `redaction.log_level`, or `redaction.mask_in_traces`; existing defaults remain `null`, `"full"`, and `false` respectively unless explicitly edited by a reviewer. `sensitivity_class` stays `"unknown"` and `scoreConfidence()` stays unchanged. This is codified as SYSTEM_SPEC.md Constraints 17 and 18, and command-surface.md documents that no new CLI command or flag is introduced.

This decision consciously rejects three tempting alternatives:

- **A) Auto-populate `retention_days` by category.** Email, payment, and secrets retention vary by product, jurisdiction, customer contract, and internal policy. A static scanner cannot choose those defaults responsibly. Rejected for V1.7.
- **B) Auto-populate `log_level` / `mask_in_traces` by category.** Logging and trace masking are runtime policy controls; field-name categories are advisory metadata. Rejected for V1.7.
- **C) Treat populated categories as regulatory compliance evidence.** A category label means "the normalized source field name matched the frozen category map." It does not prove runtime payload content, retention enforcement, GDPR/HIPAA/PCI compliance, or safe logging. Rejected explicitly under Constraint 18.

### Challenge 38: Add a reviewer aid, not a retention policy generator

VISION.md asks for redaction and retention defaults that match what reviewers actually need. The naive next step after M26 is to auto-populate `redaction.retention_days` / `log_level` / `mask_in_traces` per category (e.g., `payment → retention_days: 365`). That would overclaim: a static scanner has zero visibility into the customer contract, jurisdiction, data-residency policy, or internal risk posture that determine the right retention window. A scanner-chosen default would either be too loose (and invite a silent compliance regression) or too strict (and generate false-alarm noise the reviewer has to override case by case).

The bounded M27 step is the inverse: surface the M25/M26 evidence the reviewer needs alongside a frozen per-category reviewer reminder, and STOP there. The reviewer still owns every retention/logging/trace decision; M27 just makes the decision legible and consistent per capability. This keeps the scanner honest ("this field name matched a canonical PII name of category X, reviewer owns the rest") and avoids the silent-default-mutation risk documented under Risk #3 for M26.

**Decision:** M27 ships a read-only reviewer-aid CLI subcommand `tusq redaction review`, NOT a retention-policy generator or a default-populating command. Rejected three alternatives: (A) `tusq redaction apply-defaults` auto-populating `retention_days` per category — overclaims scanner authority, violates Constraint 18; (B) `tusq redaction recommend --write` auto-editing the manifest — conflates reviewer judgment with static analysis; (C) an implicit manifest mutation during `tusq manifest` generation that silently populates new `retention_recommendation_days` fields — creates a manifest-shape change and a `capability_digest` flip for every capability, duplicating the M26 broad-flip risk without the governance payoff. Codified as SYSTEM_SPEC Constraint 19 (reviewer-aid-framing invariant) and Constraint 20 (read-only invariant).

### Challenge 39: One new CLI noun (`redaction`) with one subcommand (`review`), not a top-level `tusq review` command

`tusq review` already has a distinct meaning under M15: it's the review-gate command that flags low-confidence/unapproved capabilities for reviewer action before compile. Grafting PII review onto `tusq review` would (a) conflate two different review concerns (approval-gate review vs PII-aware review), (b) make it easy to accidentally change `tusq review` default behavior and break existing CI callers, and (c) force M27 to grow a `--pii` flag on an existing noun that would then need to stay under the M27 advisory boundary in every future version. A separate noun `redaction` with a single `review` subcommand — the same command-group shape already established by `tusq policy` (with `init` and `verify` subcommands under M21/M22) — keeps M27 observationally invisible to every existing tusq caller that doesn't explicitly invoke it, and leaves room for a future `tusq redaction validate-wording` or `tusq redaction export-csv` subcommand without polluting any existing surface.

**Decision:** M27 introduces exactly one new CLI noun, `redaction`, with exactly one subcommand, `review`. `tusq review` continues to mean the M15 approval-gate command. `tusq redaction` (no subcommand) prints an enumerate-subcommands block identical in shape to `tusq policy`. Rejected two alternatives: (A) extending `tusq review` with a `--pii` flag — conflates two different review concerns and makes the approval gate harder to reason about; (B) adding a `pii-review` top-level command — clutters the command surface and wastes a noun slot on a narrow function that belongs under a redaction-related group. Codified as command-surface.md § M27 and documented under Constraint 19.

### Challenge 40: Freeze the advisory wording, don't parameterize it

The reviewer-advisory strings are part of the governance contract, not a localization surface.
